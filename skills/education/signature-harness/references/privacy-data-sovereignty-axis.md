# Privacy / Data-Sovereignty Axis Reference

Use this reference only when a Korean high-school SIGNATURE / SD Navigation run has **student-supplied evidence or stated interests** related to internet privacy, data sovereignty, personal data systems, local/hosted AI, network isolation concepts, OS/application isolation, memory safety, storage integrity, encryption, backup, or data governance.

This is an **optional thematic framework**, not the default SIGNATURE direction. If the student’s evidence points to biology, humanities, social policy, art/design, environment, business, medicine, or another field, build a field-appropriate framework instead.

## Evidence Gate

Before using this axis, confirm at least one of the following:

- The student explicitly says they are interested in privacy, personal data, AI data flow, servers, operating systems, security, governance, or trust.
- The record contains relevant evidence such as coding/system projects, AI ethics/data discussions, information reliability, mathematical modeling of data, storage/integrity topics, network or platform analysis, digital citizenship, or privacy-related reading/writing.
- The school task provides a performance-assessment path where data flow, ethics, policy, system design, or information reliability can be addressed safely.

If evidence is weak, label the direction as a hypothesis and ask the student before finalizing.

## Safe Center-Axis Templates

Choose or adapt one concise identity. Do not copy these mechanically if they do not match the student.

- `개인 데이터 흐름을 분석하고 보호 구조의 한계를 비교하는 정보 시스템 탐구자`
- `AI·앱·저장 과정에서 데이터가 이동·보존되는 방식을 분석하는 데이터 윤리/시스템 설계 지향 학생`
- `프라이버시와 정보 신뢰성을 함께 고려하여 디지털 서비스의 책임 있는 설계 조건을 탐구하는 학생`
- `데이터 주권을 높이기 위한 기술적·사회적 보호 구조와 trade-off를 비교하는 학생`

## Reusable Layer Model

Use a layer model only when it helps organize broad technical interests. Rename layers if needed.

```text
Layer 1. Network / Identity
- What identifiers, metadata, accounts, sessions, and access paths reveal about a user
- School-safe artifacts: data-flow map, metadata exposure table, risk/benefit comparison

Layer 2. AI / Application
- How prompts, files, logs, model memory, permissions, and app integrations move data
- School-safe artifacts: AI service data-flow diagram, privacy checklist, user guide rewrite

Layer 3. Device / OS / Runtime
- How permissions, sandboxing, virtualization, process isolation, logs, cache, and memory safety shape trust
- School-safe artifacts: isolation concept diagram, permission model comparison, memory-safety explainer

Layer 4. Storage / Integrity / Governance
- How data is stored, backed up, verified, corrected, encrypted, deleted, or governed over time
- School-safe artifacts: hash/checksum explainer, backup policy model, data lifecycle diagram
```

## School-Safe Framing

Prefer:

- `네트워크·AI·운영체제·저장 계층에서 개인 데이터가 이동·노출·보존되는 구조를 분석함`
- `보호 구조의 효과와 한계를 함께 비교함`
- `개념 모델, 체크리스트, 데이터 흐름도, 위험-대응 표를 제작함`
- `윤리, 책임, 접근성, 편의성, 성능, 비용의 trade-off를 고려함`
- `구체적 절차 안내가 아니라 원리와 한계 분석 수준에서 다룸`

Avoid in school-facing outputs:

- procedural bypass, evasion, intrusion, attack, or exploit instructions
- claims of complete anonymity, perfect security, or unconditional safety
- presenting local AI, personal servers, open source, encryption, or isolation as automatically safe
- naming sensitive tools in a way that turns the activity into a usage guide
- collecting real third-party personal data without consent

## Candidate Activity Patterns

### Data-flow map

- Purpose: show where personal data is created, transmitted, processed, logged, stored, and deleted.
- Inputs: public documentation, privacy policies, synthetic examples, or self-created toy scenarios.
- Output: diagram + limitation notes.

### Privacy checklist / rubric

- Purpose: compare services or designs using criteria such as data minimization, consent, retention, transparency, security, and user control.
- Inputs: public policy documents or fictional service scenarios.
- Output: scoring rubric + reflection on ambiguity.

### Local vs cloud processing comparison

- Purpose: compare privacy, convenience, cost, performance, reliability, and accessibility.
- Inputs: public technical docs and conceptual models, not private account data.
- Output: trade-off table + recommendation conditions.

### Isolation concept explainer

- Purpose: explain why separating roles, permissions, networks, or runtimes can reduce damage from mistakes.
- Inputs: official documentation or simplified diagrams.
- Output: concept diagram + limitations.

### Integrity / verification mini-model

- Purpose: show how hashes, checksums, versioning, backup, or error correction help detect change or loss.
- Inputs: toy strings/files, synthetic datasets, or classroom examples.
- Output: small demonstration, diagram, or report.

## Subject Connection Examples

Adapt to the student’s actual selected subjects and school performance-assessment names.

- 국어/화법/언어: privacy policy readability, public guide rewrite, debate on data responsibility.

- 과학과제연구: research question, literature review, synthetic data, model validation, limitation analysis.

## Audit Prompts

Before finalizing a privacy/data-sovereignty plan, ask:

1. Is the theme supported by the student’s record or explicit interest?
2. Does the draft avoid procedural cyber misuse?
3. Are all data examples synthetic, public, or consent-safe?
4. Does the plan discuss limitations and trade-offs instead of promising perfect protection?
5. Are unselected subjects excluded from the main roadmap?

## If the Topic Becomes Too Risky

Narrow the scope to one of these safe alternatives:

- digital citizenship and privacy literacy guide
- AI service data-flow and consent analysis
- data minimization checklist for a fictional app
- public privacy policy readability comparison
- conceptual integrity verification using toy examples
- ethics-focused debate or presentation on responsible data use
